# Motivation

As a Python-first control stack, *Astra* offers researchers and small- to medium-sized facilities a significantly faster path to autonomous operation than traditional daemon-driven systems. Whereas [RTS2](https://www.rts2.org/) offers a mature, distributed C++ ecosystem with broad driver coverage and highly robust scheduling, *Astra* emphasizes rapid deployment, minimal configuration overhead, and a modern web-native operator experience. By avoiding external service dependencies and packaging the entire observatory control stack in Python, *Astra* streamlines installation across heterogeneous computing platforms and reduces operational complexity, making autonomous observatories more accessible and ultimately promoting higher observing efficiency, improved data quality, and enhanced hardware safety.

*Astra* interfaces with observatory devices — telescopes, cameras, domes, focusers, filter wheels, safety monitors, and weather sensors — via the ASCOM Alpaca protocol. Its configuration-driven, object-oriented architecture natively supports common operational sequences — such as opening/closing, imaging runs, flats, autofocus, guiding, and pointing corrections — requiring only minor YAML/JSON edits for out-of-the-box operation. Site-specific behavior is easily introduced by subclassing the core Observatory object to override or extend algorithms, device logic, or safety policies while retaining shared services such as logging, database access, task queueing, and the web API. This design keeps standard deployments simple and robust while providing a lightweight customization path.
